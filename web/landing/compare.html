Tonet vs mainstream browsers

Tonet is intentionally narrow in scope: a transparent, policy-driven surface built around an in-repo engine path (Servo for web content on desktop). This page summarizes documented product facts today — not benchmark winners — so teams can decide fit quickly.

Dimension Tonet Google Chrome Brave Mozilla Firefox
Rendering engine (core) Servo (in-process for pages); custom shell — not Blink/WebKit/CEF. Blink (Chromium). Blink (Chromium). Gecko + SpiderMonkey.
Primary distribution model Self-hosted CDN manifest (version.json) + GitHub releases artifacts. Google update infra + OEM bundles. Brave update channels. Mozilla + distro packages.
Open-source license (browser) GPL-3.0-or-later (repository). Chromium BSD-style + proprietary Chrome additions. MPL/Chromium mix; Brave-owned services. MPL 2.0.
Default web compatibility target MVP reading/search stack; strict page-size policy (e.g. 1 MB guard). Full modern web. Full modern web + shields. Full modern web.
Extension ecosystem Not a goal for MVP (no Web Store parity). Chrome Web Store. Chrome-compatible extensions. Firefox AMO + MV3 transition.
Update check mechanism (Tonet app) Configurable manifest URL; no GitHub Releases API dependency at runtime. Google updater. Brave updater. Mozilla updater.
Typical install size (order of magnitude) Single-digit ×10 MB binary + platform deps (Windows ANGLE DLLs alongside exe). Hundreds of MB installed. Similar to Chromium-class. Hundreds of MB installed.

Rows reflect architecture and delivery choices described in this repository as of the current milestone — not marketing claims or independent benchmarks. Refresh this table when major milestones ship (see Roadmap).

When Tonet is a strong fit

  • You need audit-friendly internals and deterministic packaging over maximal site compatibility.
  • You prefer CDN-owned artifacts and manifests over opaque auto-update channels.
  • You accept MVP limits today in exchange for an explicit roadmap (TONET_VISION.md).
